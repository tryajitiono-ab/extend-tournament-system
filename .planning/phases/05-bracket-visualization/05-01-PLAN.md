 * @param {string} namespace - Namespace ID
 * @param {string} tournamentId - Tournament ID
 * @returns {Promise<Object>} Object with matches, totalRounds, currentRound
 */
async function fetchMatches(namespace, tournamentId) {
    const url = `${API_BASE}/v1/public/namespace/${namespace}/tournaments/${tournamentId}/matches`;
    const response = await fetchWithTimeout(url);
    
    if (!response.ok) {
        throw new Error(`Failed to fetch matches: ${response.statusText}`);
    }
    
    const data = await response.json();
    return {
        matches: data.matches || [],
        totalRounds: data.total_rounds || 0,
        currentRound: data.current_round || 0,
    };
}

Create new bracket-adapter.js module that transforms REST API match data to brackets-viewer.js format.

Key transformation requirements from RESEARCH.md:

**Status mapping:** Our protobuf enum → brackets-viewer numeric status
- MATCH_STATUS_SCHEDULED → 2 (Pending)
- MATCH_STATUS_IN_PROGRESS → 3 (Running)  
- MATCH_STATUS_COMPLETED → 4 (Completed)
- MATCH_STATUS_CANCELLED → 5 (Archived)

**Round indexing:** Our API uses 1-indexed rounds (Round 1, Round 2), brackets-model uses 0-indexed
- Transform: round_id = match.round - 1

**Participant mapping:** 
- id: participant.user_id (not participant_id)
- name: participant.username || participant.user_id (fallback for missing username)

**Match structure:** brackets-viewer expects:
- opponent1/opponent2 objects with { id, position } properties
- null opponents for BYE matches or unknown participants
- position calculated from match position (opponent1: position*2-1, opponent2: position*2)

**Output structure:**
```javascript
{
  stages: [{ id: 0, tournament_id, name, type: 'single_elimination', number: 1 }],
  matches: [{ id, stage_id: 0, group_id: 0, round_id, number, opponent1, opponent2, status }],
  participants: [{ id, tournament_id, name }],
  matchGames: []
}
```

Implementation:
1. Create transformToBracketsModel(matches, participants, tournament) function
2. Create mapMatchStatus(apiStatus) helper for status enum conversion
3. Handle null/undefined participants gracefully (BYE matches, unknown slots)
4. Add JSDoc comments for all exported functions
5. Handle empty matches array (tournament not started)

Error handling:
- Return valid empty structure if matches is empty array
- Don't throw errors - graceful degradation
- Log warnings for unexpected data shapes

Follow Phase 4 patterns:
- Single-file module (no imports/exports keywords)
- Functions available in global scope
- Document dependencies in comments (uses Match[], Participant[], Tournament from API)
