## Issues Found During Testing

### Issue #1: Winner Advancement Bug 🐛
**Severity:** CRITICAL - Blocks tournament completion
**Symptom:** Player 4 didn't advance to finals after winning semi-final

**Root Cause:**
- `calculateNextPosition()` used wrong formula for 0-indexed positions
- Formula: `(position - 1) / 2 + 1`
- Position 0 → Position 0 ✅ (correct by accident)
- Position 1 → Position 1 ❌ (should be 0)
- Position 2 → Position 1 ✅ (correct)
- Position 3 → Position 2 ❌ (should be 1)

**Fix Applied:**
```go
// OLD (WRONG)
func calculateNextPosition(currentPos int32) int32 {
    if currentPos <= 0 {
        return 0
    }
    return (currentPos-1)/2 + 1  // Brittle formula
}

// NEW (CORRECT)
func calculateNextPosition(currentPos int32) int32 {
    return currentPos / 2  // Simple integer division
}
```

**Test Result:** ✅ Both winners now correctly advance to finals!

- `pkg/service/match.go` (lines 107-112)

### Issue #2: No Round Completion Safeguard 🐛
**Severity:** HIGH - Data integrity risk
**Symptom:** Could submit Round 2 results before Round 1 complete

**Problem:**
- No validation to enforce sequential round completion
- Could create invalid tournament states
- Winners might advance before previous matches finish

**Fix Applied:**
Added validation in both submit methods:
1. Check if target match is in round > 1
2. Verify ALL matches in previous rounds are COMPLETED
3. Return `FailedPrecondition` error if not

```go
// Check if all matches in previous rounds are completed
for round := int32(1); round < targetMatch.Round; round++ {
    previousRoundMatches, err := m.matchStorage.GetMatchesByRound(ctx, req.Namespace, req.TournamentId, round)
    // ... check each match status ...
    if prevMatch.Status != serviceextension.MatchStatus_MATCH_STATUS_COMPLETED {
        return nil, grpcStatus.Errorf(codes.FailedPrecondition,
            "cannot submit result for round %d: match %s in round %d is not completed",
            targetMatch.Round, prevMatch.MatchId, round)
    }
}
```

**Test Result:** ✅ Cannot submit finals result before Round 1 complete!

## Architecture Issues Identified

### Known Issue: Brittle Match Progression Architecture 🏗️
**Severity:** TECHNICAL DEBT - Future maintainability
**Status:** DOCUMENTED (not fixed in this session)

**Problem:**
Current system uses **position mathematics** to figure out where winners advance:
- Relies on formula: `nextPosition = currentPosition / 2`
- Works but is BRITTLE and hard to understand
- No explicit parent/child relationships between matches
- Difficult to extend for other bracket types (double elimination, swiss, etc.)

**User Feedback:**
> "I don't know exactly how the tournament service is deciding how players progress through rounds but I saw calculateNextPosition and it is just some formula if this is truly the behaviour and this is BRITTLE AF!"

**Better Architecture (Future Refactor):**

Instead of position math, use explicit match relationships:
```go
type Match struct {
    MatchId        string
    NextMatchId    string  // Where does winner go?
    SourceMatch1Id string  // Where did participant1 come from?
    SourceMatch2Id string  // Where did participant2 come from?
    // ... other fields
}
```

**Benefits:**
- ✅ Clear, explicit relationships
- ✅ Easier to understand and maintain
- ✅ Flexible for different bracket types
- ✅ No brittle math formulas
- ✅ Self-documenting code

**Recommendation:**
Plan this refactor for v1.3 or v2.0. Current math-based approach works correctly after fix but is technical debt.

**Task Created:** #14 - Refactor bracket progression to use explicit match relationships

## Known Issues (Not Fixed)

### BYE Advancement Too Aggressive
**Status:** EXISTING BUG (not introduced by this fix)
**Severity:** MEDIUM
**Symptom:** Finals auto-completed with player-1 before player-4 arrived

**Root Cause:**
`HandleByeAdvancement()` runs after each match result and auto-completes any match with only ONE participant (treating missing opponent as BYE).

**When It Happens:**
1. Player 1 wins Match 1 → advances to finals
2. BYE handler sees finals has only 1 participant
3. Auto-completes finals with player-1 as winner
4. Player 4 wins Match 2 → tries to advance but finals already complete

**Workaround:**
Clear matches collection between tournament tests.

**Proper Fix (Future):**
Only run BYE advancement:
- At tournament start (for pre-seeded BYEs)
- When explicitly triggered by admin
- NOT automatically after every match result

**Decision:** Not fixing in this session - separate issue to address later.
